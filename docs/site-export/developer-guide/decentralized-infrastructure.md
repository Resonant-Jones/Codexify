# Decentralized Infrastructure

Codexify’s longer direction is local-first sovereign cognitive infrastructure for a decentralized network.
That direction should be read as product and architecture intent built from current primitives, not as a claim that full federation or autonomous multi-node operation is already shipped.

## Local-first runtime

The current supported path already starts from a local node model.
Compose-local services, local-only provider posture, and explicit operator health surfaces create a runtime that can function without a required central coordinator.

## Export and restore portability

Account export and restore are sovereignty guarantees.
Portable archives with manifest, lineage, relationships, and integrity metadata are the basis for user-controlled migration, backup, and eventual cross-node continuity.

## Lineage preservation

Codexify’s contracts repeatedly preserve authored-turn identity, request-attempt identity, command lineage, and artifact provenance.
That discipline matters for decentralized systems because portable state without lineage becomes untrustworthy state.

## Bounded extension governance

Codexify’s extension doctrine is not “self-modify at will.”
Proposal, forge, sandbox, review, registration, runtime binding, and rollback metadata are the governance seams that make extensibility compatible with sovereignty.

## Provider governance

Configured provider, discovered inventory, supported profile, and live runtime health are treated as separate truths.
That separation is a useful decentralized primitive because execution authority stays explicit instead of being inferred from a remote provider’s mere presence.

## Retrieval doctrine

Retrieval widening belongs in orchestration, remains user-bound, and starts narrow.
That doctrine is compatible with future peer or federated retrieval because it forces scope, source mode, and widening reason to be explicit before cross-boundary evidence is considered.

## Future federation boundaries

The repo already treats federation and peer context as explicit trust-policy surfaces rather than ambient connectivity.
A future decentralized network would need to preserve:

- node trust boundaries
- user-bound retrieval and identity scope
- capability-based access instead of ambient authority
- versioned contracts for sync, lineage, and restore
- failure-tolerant queue, retry, and observability behavior across partitions

## Not Yet Shipped

The following should not be described as shipped product reality:

- general federation as part of the supported beta path
- durable cross-restart sync subscriptions
- cloud-provider support as part of the supported local-first contract
- autonomous recursive coding-agent execution
- direct Pi-like harness execution
- graph-write behavior as part of the default supported Compose path
- decentralized network coordination beyond current bounded contracts and experimental surfaces
