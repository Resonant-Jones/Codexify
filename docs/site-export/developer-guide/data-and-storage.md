# Data and Storage

Codexify keeps durable state across several storage systems, each with a distinct role.

## Storage Systems

- Postgres stores the canonical application record.
- Redis stores coordination state such as queues, locks, and short-lived task visibility.
- The vector store holds semantic retrieval data.
- File or object storage holds raw uploaded or generated bytes and artifacts.
- Optional Neo4j holds graph-related data when graph features are enabled.
- Browser local or session storage holds client-side state such as runtime overrides and UI session data.

## Key Entities

The important persistent entities are the ones that preserve lineage and restoreability:

- projects
- chat threads
- chat messages
- uploaded and generated documents
- media assets
- command runs
- cron runs
- audit and domain events

Other entities matter because they preserve ownership, scope, or provenance around those core records.

## Lineage Rules

Lineage is not decorative metadata.
It is what makes export, restore, replay, and debugging trustworthy.

The current contracts require:

- stable identifiers where restore depends on them
- explicit relationship records where joins matter
- preserved source provenance for imported or derived material
- explicit restore failure when fidelity cannot be maintained

## Restore Obligation

Restore is not allowed to silently drop:

- ownership
- project membership
- thread or message ordering
- artifact relationships
- provenance
- dedupe identity

If a restore cannot preserve one of those, it must fail closed or report the loss explicitly.

## Operational Meaning

Storage is part of the runtime contract, not a passive implementation detail.
When storage drift occurs, the operator-facing truth surfaces and the current-state docs matter as much as the schema itself.
