# Zac Report Archive

**For:** Zac and Zac's agent  
**Last updated:** 2026-06-26  
**Status:** Active — a bounded report archive, not a task backlog

## Purpose

This folder stores report-only learning artifacts generated by Zac or Zac's agent. Reports help Zac build a mental map of Codexify over time — one area, one lens, one observation at a time.

Reports are allowed to be exploratory, partial, and question-driven. They exist to build familiarity, not to justify implementation. A report that raises more questions than it answers is a successful report.

Reports are not implementation tasks. They are not proposals. They are not architecture authority. They are learning artifacts that may or may not lead to future work.

## What Belongs Here

- Directory maps (Cartographer lens).
- UI observation reports (UI Naturalist lens).
- Documentation orientation reports (Doc Gardener lens).
- Runtime boundary reports (Runtime Boundary Scout lens).
- Dev-experience reports (Dev-Experience Mechanic lens).
- Test coverage reports (Test Cartographer lens).
- Continuity phase orientation reports (Continuity Museum Guide lens).
- Other report-only learning artifacts that follow the report templates in `../report-output-templates.md`.

## What Does Not Belong Here

- Codexify task prompts.
- Implementation instructions or code patches.
- Approved architecture changes or ADR amendments.
- Release claims or supported beta assertions.
- Backlog items, tickets, or work assignments.
- Unreviewed claims presented as current truth.
- Secrets, credentials, API keys, or personal data.
- Proposals — those belong alongside `../proposal-template.md` in Zac's main directory.

## Report Statuses

Each report in this archive should carry one status in the `report-index.md` table:

| Status | Meaning |
|---|---|
| `draft` | Generated but not yet reviewed by Zac or Resonant. |
| `reviewed` | Read by Zac or Resonant. No immediate action identified. |
| `proposal-needed` | Report identified something worth proposing later. A proposal should be written separately using `../proposal-template.md`. |
| `archived` | Useful for historical context. No action planned and no longer actively referenced. |
| `superseded` | Replaced by a newer report, updated docs, or current-state truth. The old report is kept for context but should not be treated as current. |

## How To Add A Report

1. **Choose one report lens** from `../report-only-agent-lenses.md`.
2. **Generate one report only** using the prompts in `../report-request-prompts.md`.
3. **Save the report** as a markdown file in this folder. Follow the filename conventions in `report-filename-conventions.md`.
4. **Add an entry** to `report-index.md` with date, report title, lens, area, status, key question, and follow-up notes.
5. **Keep evidence paths in the report.** Cite specific files, docs, tests, or live behaviors. Do not make claims without references.
6. **Do not include implementation instructions** unless the report is later explicitly converted into a proposal elsewhere.

## Authority Boundary

Reports are learning artifacts. They have no authority over Codexify's architecture, release truth, or implementation.

- Reports do not override `docs/architecture/00-current-state.md`.
- Reports do not override ADRs or architecture contracts.
- Reports do not authorize implementation.
- Reports do not create tasks or backlog items.
- Reports do not imply Resonant approval.
- Reports do not claim runtime truth or supported beta behavior.

If a report and `00-current-state.md` disagree, `00-current-state.md` wins. If a report identifies a genuine conflict, that conflict should be raised as a question for Resonant — not silently treated as resolved.
