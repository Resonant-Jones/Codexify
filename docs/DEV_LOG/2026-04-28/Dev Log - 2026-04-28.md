What I completed

Auth and startup trust boundary

- Removed the predictable local bootstrap credential while keeping the canonical single-user owner as `local`.
- Added `GUARDIAN_BOOTSTRAP_PASSWORD` as the explicit escape hatch; when unset, fresh instances now fail closed instead of shipping a guessable login.
- Mirrored the same behavior into the migration path so old databases do not reintroduce the insecure seed.
- Added focused auth/identity regression coverage and kept the config surface truthful in `.env.example`.

Why it matters: the system still has a local owner, but it no longer treats a default password as part of the trust model.

Chat source vocabulary

- Kept `project` as the default posture and normalized legacy readback so persisted data stays compatible.
- Verified the change with focused tests where the workspace harness allowed it, and accepted the harness mismatch as an environment issue rather than pretending it was a product regression.

Why it matters: the chat UI now says the same thing the backend actually means.

Packaged runtime and desktop behavior

- Tightened desktop auth header precedence.
- Fixed packaged desktop auth handoff and packaged live events auth transport.
- Fixed packaged Guardian thread creation and exposed packaged runtime health diagnostics in Guardian.
- Kept the runtime-health and thread-creation paths aligned with the actual packaged environment instead of only the repo-local one.

Why it matters: the desktop shell is less likely to drift into a state where it looks healthy but cannot actually start or connect cleanly.

Capability activation and dispatch

- Added a bounded capability activation decision seam.

Why it matters: activation is now a concrete contract with explicit allow/deny outcomes, not a hand-wavy future feature.

Graph lane visibility

- Mounted a bounded no-op graph backend adapter behind it without changing runtime behavior.
- Kept the lane inspection-only and non-canonical, with TTL-based snapshots and explicit empty-state behavior.

Why it matters: graph writes now have a disciplined control/inspection plane without pretending persistence exists yet.

Tool jobs proof

The system stopped relying on implied behavior and started naming its boundaries.

The graph lane is still not canonical storage, but it now has structured visibility instead of silent ambiguity. That is a different kind of maturity: less magical, more defensible.

Narrative Log

Dev Log - 2026-04-28

The day read like a long correction pass across the system’s trust boundaries. It started with a simple but important fix: fresh Codexify instances should not ship with a password that anyone can guess. The bootstrap path was rewritten so `local` remains the canonical owner, but the login secret now comes from an explicit operator override or a one-time random secret. That change is small in code and large in meaning. It turns a hidden assumption into an intentional setup choice.

From there, the work widened into the chat and runtime surfaces that had been carrying outdated language or brittle handoff logic. The backend `workspace` source was wired into the live retrieval contract, then the frontend was updated to say `workspace` instead of `personal_knowledge` while still normalizing old persisted values. In the same general area, the packaged desktop path got a series of fixes around auth headers, live events transport, thread creation, and runtime health reporting. The point was not cosmetic polish. It was to make the packaged app behave like the real runtime, not a hopeful approximation of it.

The extension layer moved in the same direction: activation became explicit, bounded, and lineage-aware. Rather than implying that capability dispatch might happen someday, the code now expresses an activation decision, a manual dispatch envelope, and a command-bus-shaped invocation with provenance threaded through it. That keeps the system honest about what is allowed, what was resolved, and what actually ran. The graph lane followed the same pattern. It is still inspection-only and non-canonical, but now it has a thread-scoped snapshot surface and a no-op adapter seam that can be swapped later without pretending persistence exists today.

The other piece worth naming is the `tool_jobs` proof. That was not another speculative seam. It was a live Compose-path verification artifact for a narrow cleanup migration, and it was added to the architecture map as evidence. That distinction matters. Most of the day was about tightening code and contracts; that one piece was about documenting runtime truth that had actually been observed.

What changed underneath all of this is simple: the system is becoming harder to drift. Identity, retrieval source, activation, and graph visibility are all more explicit now. There is less room for silent fallback and less room for a label to mean one thing in one place and something else in another. That is the kind of progress that does not look dramatic from the outside, but it makes everything else more reliable.
