Summary

Startup-readiness and shell-posture hardening day.

Most of the work was about making the desktop launcher tell the truth about why the local runtime is or is not available. The first-run path moved from a generic bootstrap state into an explicit readiness contract, and the recovery UI now carries that state through to the app. A second thread cleaned up the Documents shell so the sidebar can be dismissed without trapping the center lane, including a mobile overlay path and a desktop edge affordance.

What I completed

Launcher setup readiness
- The first-run wizard now classifies setup into explicit states instead of collapsing everything into a generic blocked state: missing or incomplete config, Docker/Ollama availability, selected model presence, Compose validity, existing volumes, backend health, frontend availability, and ready.
- The launcher startup handoff now carries a `setupReadiness` snapshot into the desktop app, so the recovery screen can explain the actual blocker instead of repeating the same vague failure.
- Retrying setup now refreshes the routing decision rather than just flipping a UI phase flag, so the app re-reads launcher state before deciding whether to proceed.
- Supporting env defaults were aligned so the desktop handoff points at the expected runtime endpoints.
- The readiness matrix and runtime-config normalization path are covered by tests.

Failure classification
- Native bridge/import failures are now separated from Docker preflight failures.
- A broken Tauri bridge no longer gets mislabeled as "Docker Desktop is required"; the app can now distinguish "can’t talk to the desktop runtime" from "Docker is actually missing."
- The bootstrap mapping and tests were updated so the existing Docker guidance still appears when Docker is the real issue.

Documents shell posture
- The Documents sidebar can now be dismissed on desktop, re-opened from an edge affordance, and shown as an overlay on mobile.
- When workspace focus is active on constrained widths, the sidebar auto-collapses so the center lane stays usable.
- The new posture keeps Documents from duplicating itself with the workspace drawer, and Guardian shell behavior stays untouched.
- The AppShell and DocumentsView tests were expanded to lock that posture in.

Important outcome

The launcher is no longer just checking whether "something" failed. It can now name the failure class, carry that state into the recovery UI, and ask for the right next action. That is a meaningful shift from ambiguous gating to a structured handoff.

The Documents shell also stopped behaving like a fixed rail. It can get out of the way without losing the center lane, which makes the shell feel less trapped and more deliberate. This is still contract-level truth, not live proof, but it is a cleaner basis for the next round of runtime verification.

Dev Log - 2026-04-26

Launcher readiness

This was a truth-alignment day more than a feature day. The biggest job was to make the desktop startup path distinguish between different kinds of failure instead of flattening them into one blocked state. The setup wizard now classifies readiness across config, Docker, Ollama, model presence, Compose validity, local volumes, backend health, and frontend health. That gives the launcher something concrete to hand off instead of a vague binary.

Once that state existed, the app could stop pretending the recovery path was generic. The recovery screen now carries the readiness details through the launcher handoff, and retrying setup re-reads the launcher state instead of just toggling UI. I also aligned the example environment defaults so the desktop handoff points at the expected runtime endpoints. The result is not just cleaner startup messaging; it is a more honest startup contract.

Failure boundaries

The next correction was smaller but important: a broken native bridge should not look like a missing Docker install. I split native bridge/import failures from Docker preflight failures so the app can point at the right layer. That matters because the remediation is different. If the runtime bridge is broken, the answer is to open Codexify from the desktop app and retry. If Docker is actually absent, the answer is different. The system should not send the operator down the wrong path.

The launcher work also got the usual support scaffolding: tests around the readiness matrix, tests around runtime-config normalization, and the architecture ops doc updated to describe the new first-run states. That keeps the behavior from drifting back into vague startup language later.

Documents posture

The other half of the day was the Documents shell. The sidebar is no longer a permanent obstruction. It can collapse on desktop, overlay on mobile, reopen from an edge affordance, and auto-collapse when workspace focus is active on a constrained viewport. The center lane stays usable, which is the real boundary condition. The shell is starting to behave like a posture system instead of a fixed rail.

That mattered because the old shape was too easy to trap yourself inside. Now the surface can get out of the way without losing the document view itself, and the tests were updated to keep that posture intact. Guardian shell behavior stayed untouched, which is the right outcome for a fix like this. The change stays local to the Documents experience instead of leaking into the rest of the shell.

What this makes possible

The broader shift is simple: the app is getting better at naming its own state. Not just "working" or "broken," but which part is missing, which path is unavailable, and which UI should step aside. That makes the next live proof pass more meaningful because the contract is already clearer.
