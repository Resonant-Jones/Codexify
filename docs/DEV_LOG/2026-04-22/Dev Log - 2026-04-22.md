## Dev Log - 2026-04-22
**Summary**

contract hardening and runtime truth day

The day was mostly about making Codexify more honest about what is supported and what is still only a contract or proof artifact. The backend gained durable observability for bounded tool turns and a post-completion eval spine, while the frontend was pared back to the real beta surface instead of demo-friendly leftovers.

The other thread was identity and continuity. Verified personal facts were made a bounded, user-scoped context source, ChatGPT custom-instruction facts were extracted before filtering, and long values were normalized so they fit storage. Several architecture ADRs were added to pin down Flow Builder threading, continuity governance, and capability resolution precedence without overstating runtime behavior.

**What I completed**

**Runtime observability and prompt truth**
- Persisted tool-loop observability fields into finished-run metadata and `task.completed` payloads so the worker does not need to stay alive just to reconstruct the tool-turn boundary.
- Added the post-completion eval spine: durable trace snapshots, deterministic groundedness scoring, a queued eval worker, and a read-only `/api/chat/debug/evals/{thread_id}/latest` route.
- Added a current-turn priority block to the system prompt so the assistant is explicitly told to answer the latest message first.
- Fixed the schema lineage around eval and extension-install migrations so the migration graph stays coherent.

**Identity and personal facts**
- Verified active personal facts now enter provider-ready context as a bounded block, with IDs surfaced in diagnostics and candidate/disputed/inactive entries excluded.
- The ChatGPT migration path now extracts third-person `model_editable_context` facts before filtering, and fact values are truncated to the DB column width before insert so long custom-instruction text does not fail silently.
- The settings surface now separates imprint review from personal-facts controls instead of bundling them together.

**UI and shell cleanup**
- Removed unfinished beta surfaces from primary nav.
- Tightened dashboard recent-thread and gallery behavior to remove demo fallbacks and cap visible items, right-anchored the documents workspace, and removed the nested shell from Documents.
- Centered the settings panel with a sticky tab rail and a cleaner edge chrome contract.

**Architecture contracts and docs**
- Added ADR-012 through ADR-015, covering the eval spine, verified personal facts, Flow Builder thread/draft/receipts, and the future continuity engine. These are accepted contracts, not claims that the runtime already ships those features.
- Added the effective capability resolution seam with profile > project > account precedence and fail-closed ambiguity handling.
- Fixed the Flow Builder doc links, refreshed the current-state doc and live supported-path proof artifact so the KB reflects runtime truth, and added the missing extension-install bindings migration.
- Simplified the local research ignore path.

**Important outcome**

The biggest shift is that the system stopped depending on transient worker memory to explain itself. Tool-loop metadata now survives into durable finished-run records, eval snapshots can be inspected after completion, and the prompt assembly path is stricter about the current turn. That makes the runtime easier to reason about and less likely to drift silently.

Verified personal facts also became structurally honest instead of loosely inferred. The backend now distinguishes verified context from candidate or disputed material, the ChatGPT import path no longer drops third-person custom-instruction facts before extraction, and oversized fact values are trimmed to fit storage instead of failing the import path. The seam is narrower and more trustworthy now.

The other important shift is the negative proof. The live supported-path rerun on the exact current `main` tip still failed, and it failed in the ways that matter: unsupported local-only posture, missing `agent_extension_*` tables on upload, and bounded tool-loop cases collapsing into plain answers. The docs now say that plainly, which matters because it leaves no room for pretending the runtime is healthier than it is.

## Narrative Log - 2026-04-22
This was a day of subtracting false signals before adding anything new. The visible interface was trimmed first: unfinished beta surfaces came out of the primary nav, the dashboard stopped leaning on demo gallery behavior, Documents lost its nested shell, and Settings was re-centered around the surfaces that are actually meant to stay. That kind of cleanup does not make headlines, but it matters because it narrows the product to the parts we can support honestly.

The backend work followed the same pattern. Tool-loop observability stopped living only in transient worker state and was written into finished-run metadata instead, which means post-run inspection no longer depends on catching a worker in the act. On top of that, the post-completion eval spine landed: durable trace snapshots, a deterministic groundedness check, a best-effort eval worker, and a route for reading the latest eval diagnostics. The system prompt also got a sharper current-turn priority block, which is a small change with real implications for how much older context can pull the conversation off course.

Identity and import handling were tightened at the same time. Verified personal facts now enter the provider-ready context path in a bounded way, with the broker surfacing exactly which fact ids were included and the prompt builder labeling the block as verified. The ChatGPT import path learned to extract third-person `model_editable_context` facts before that content is filtered away, and it now truncates long fact values so the database column width does not silently kill persistence. Alongside that, the capability-resolution seam was made explicit, and the architecture docs were expanded to pin down Flow Builder threading, continuity governance, and the line between contracts and live behavior.

The last part of the day was the most useful because it refused to flatter the work. The supported-path proof was rerun on the exact current `main` tip, and it still failed. The runtime is not in the supported local-only posture, upload still dies on missing `agent_extension_*` tables, and the bounded tool-loop cases still collapse into plain answers. That is not a pleasant result, but it is the right one to carry forward, because the docs now reflect the gap instead of hiding it. It leaves the next pass with a clear job: fix the runtime truth first, then prove it again.
