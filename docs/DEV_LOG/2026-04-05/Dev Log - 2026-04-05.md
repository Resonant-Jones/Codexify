Import enrichment decoupling
- ChatGPT import now persists the canonical rows first, marks graph work pending, and hands embedding catch-up to `chat_import_embed` queue work.

- Test fixtures were updated to drain the new queue between cases, and migration tests now validate queue handoff instead of pretending import completion depends on inline embedding.

Important outcome

Codexify stopped conflating canonical import with enrichment. Import success now means durable Postgres state first, and enrichment is a separate concern that can be retried, degraded, or deferred without invalidating the import itself.

The chat surface also got more honest about its own state. Slow responses now present a canonical lifecycle instead of a vague waiting mode, and oversized user input no longer gets to reshape the shell. The system is tighter at the edges, which is where a lot of real trust is won.

Narrative Log
Dev Log - 2026-04-05

Today was a boundary day. The work was not about adding a new user-facing feature so much as making the system say the right thing about itself. The morning audit and current-state override were refreshed first, which mattered because the rest of the changes needed a truthful top line: Codexify is still in supported-local-beta hardening, not a finished delegation product.

From there, the chat runtime was tightened in two different ways. The slow-path state hook stopped improvising its own language and started deriving a canonical lifecycle snapshot. That gave the UI a single answer for queued, warming, waiting, streaming, degraded, error, completed, and cancelled states, and it let the message region carry that truth as attributes instead of guessing from layout state. In parallel, the shell stopped letting oversized user messages define the width of the conversation. Long pastes now collapse by default, can be expanded locally, and are constrained so they do not destabilize the entire chat pane.

The largest shift was on import enrichment. ChatGPT imports now do the durable thing first: write canonical rows, mark graph work pending, and hand embedding catch-up to a queue. The worker then owns the expensive follow-up and reuses its vector store instead of forcing the import path to pay that cost. That separation is the right one. If the queue is unhealthy, import still lands and enrichment degrades explicitly. If the worker is healthy, the backlog drains without entangling the request lane.

This is the kind of work that makes a system easier to trust under partial failure. The docs now describe reality more accurately, the chat UI reports lifecycle truth more consistently, and the import path no longer couples durability to enrichment latency. The next step is to prove this shape in a live runtime and decide how far the delegation backbone is meant to go.
