## Dev Log - 2026-04-18

### Summary
Contract hardening day. The work was about taking two places where Codexify had been implicit and making them named, inspectable, and harder to drift: packaged first launch now writes its own startup-state record, and frontend tool actions now enter through the command bus instead of the legacy shim shape.

That matters because these are the kinds of boundaries that hide failure when they stay soft. The launcher can now explain its own bootstrap state, and the write path from the frontend to the backend is narrower and more honest. The changes are still code/test-level, but they reduce ambiguity in both operational state and command routing.

### What I completed

#### Packaged launcher first-launch state
- The packaged launcher now materializes a startup-state file during first launch, alongside the runtime manifest and marker.
- Bootstrap resolution now emits structured logs for app-support resolution, runtime-root and resource-root discovery, and the manifest/marker/startup-state writes.
- The startup path was refactored through a helper so runtime resolution and environment priming happen before the app is managed.
- A unit test now proves the startup-state file is serialized and written to the expected Application Support location.

#### Command bus tool-action routing
- Frontend tool execution now invokes the command bus directly and keeps a local job snapshot cache instead of treating every tool action as a legacy shim job.
- `useTriggerAction` now resolves completed command-bus responses immediately and rejects blocked or denied outcomes explicitly.
- Guardian chat profile switching now uses the canonical command-bus invoke path, and the backend exposes a dedicated profile-switch patch route with an explicit policy allowlist.
- Focused tests were added around the command-bus profile-switch path and the frontend trigger-action behavior, so the seam is now exercised where the old shim used to be implied.

#### Docs and drift cleanup
- The legacy tools inventory and risk notes were updated to mark the two live frontend callers as migrated and to narrow the remaining risk to shim coexistence rather than active frontend dependence.
- The docs now reflect the code change instead of overstating the shim as a live frontend dependency.

### Important outcome
The packaged app is no longer treating first launch as a silent internal side effect. It now leaves behind a concrete startup record, which matters when you need to inspect an installed build instead of inferring bootstrap state from logs alone.

The tool surface moved the same way. Profile switching and trigger actions are now named command-bus flows instead of ad hoc calls into the old `/tools` shape, which gives the system a clearer write boundary and fewer places for contract drift to hide.

## Narrative Log - 2026-04-18

Today was about making the system stop hand-waving at two important boundaries. The first was packaged startup: the launcher used to finish materializing the runtime without leaving a durable startup record in Application Support. The second was tool execution: the frontend still had one foot in the legacy shim, which made a controlled write look like a generic tool call.

The launcher change did not make the app prettier. It made it more legible. First launch now leaves behind a startup-state file, and the bootstrap path logs the root locations and artifact writes that matter when something goes wrong after packaging. That is the kind of work that does not show up as a feature, but it changes how support and postmortem work happen because the system now writes down what it did.

The command-bus work fixed a different kind of ambiguity. Tool actions are now going through the command bus directly, with the frontend treating completed, failed, blocked, and denied states explicitly instead of collapsing them into one old-shim shape. Profile switching became a named mutation rather than a generic tool call, and the backend policy had to acknowledge that by allowing that write intentionally instead of by accident. The shim inventory and risk notes were updated too, because drift is not only a code problem; it is also a documentation problem when the docs keep describing a dependency that is no longer active.

The deeper shift is simple: the system is moving from implicit side effects to explicit state and named writes. That gives operators a cleaner story when things work, and a shorter path to the cause when they do not. The next step is live proof on the current tip and a cleanup pass on the stale current-state note.
