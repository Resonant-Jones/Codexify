Dev Log - 2026-04-13

Summary

identity and capability hardening day

Today was mostly about turning implicit ownership and capability assumptions into explicit runtime contracts. The work narrowed the system in the right places: account scope now governs access across the main resource routes, capability grants have canonical tokens plus a real issuance path, and Persona Studio gained an isolated chat harness that cannot leak into live state.

That matters because the day did not add surface area so much as it removed ambiguity. The runtime is easier to reason about, and the next layer can attach to clearer boundaries.

What I completed

Request scope ownership hardening
- Chat, document, project, and media surfaces now resolve ownership through `RequestUserScope` and reject cross-account access in multi-user mode.
- Single-user behavior stays intact, which keeps the change boundary-focused instead of rewriting legacy flows.
- New tests cover owned-resource access, cross-account rejection, and the single-user fallback path.

Capability grant substrate
- Added canonical capability-grant tokens for scope, kind, status, and capability family.
- Added normalized grant resolution so active grants are filtered by status, time bounds, and account ownership instead of ad hoc checks.
- Added an internal grant issuance path plus CLI plumbing to mint durable grants against known tiers with provenance and time-window validation.
- Added contract tests for issuance normalization and fail-closed behavior when inputs are invalid or non-canonical.

Phase-0 policy framing
- Added a phase-0 execution-kernel prep spec that explicitly keeps this phase in current-runtime hardening.
- The spec locks the next layer to canonical runtime vocabulary, observable request state, explainable retrieval posture, and preserved provenance.
- That is not shipped orchestration work; it is guardrail work that keeps future runtime expansion from drifting the current contract.

Persona Studio
- Added an ephemeral chat harness that exercises a persona draft without creating Guardian threads or writing session state.
- The harness keeps draft snapshots visible so later replies reflect edits in a way users can inspect.
- Tests cover multi-turn behavior, session reset, draft-change awareness, and the guarantee that the harness stays off runtime write paths.

Runtime startup hardening
- Lazy-loaded `guardian.core.db` on first access so backend adapter selection happens at use time instead of import time.
- That reduces startup brittleness and keeps the database boundary less eager.

Important outcome

The system stopped treating ownership and grant eligibility as background assumptions. It now has explicit account-scope checks in the main resource routes and explicit canonical grant tokens in the capability layer, which makes the runtime harder to misuse and easier to audit.

Persona Studio also became more truthful. It is no longer pretending to be a live assistant surface when it is really a draft sandbox; the ephemeral harness makes that separation explicit and keeps the live system clean.

Narrative Log

Dev Log - 2026-04-13

Today was a boundary day, and the boundary work is the part that matters. The biggest shift was not visible in a feature checklist; it was in the way the runtime now names its own limits. Chat, documents, projects, and media stopped relying on ambient access and started deriving ownership from request scope. That change is small in code shape and large in system meaning: the multi-user path now fails closed, while the single-user path still behaves the way it did before.

The same pattern showed up in the capability layer. Canonical tokens, normalized grant resolution, and an internal issuance path moved capability handling out of the realm of informal policy and into something the runtime can actually enforce. The work is still foundation, not rollout. The point of it was to make capability eligibility concrete enough that later policy code can stand on it without inventing new vocabulary.

Persona Studio went in the other direction, but for the same reason. The ephemeral chat harness does not make the surface more powerful in the live sense; it makes it more honest. It lets a persona draft be exercised in a temporary conversation without creating Guardian threads or touching persistence, which means the design surface can be useful without pretending to be the production runtime. That separation matters. A draft editor that leaks into live state is a bad boundary. This one now makes the boundary visible.

The startup hardening was quieter but in the same family. Lazy-loading the DB adapter removes one more place where import-time behavior could surprise the process before the runtime was ready to choose its backend. It is the kind of change that reduces failure noise rather than adding capability, and that is the right trade today.

What changed underneath the surface is straightforward: the system is becoming more explicit about who owns what, which grants are real, and which surfaces are only for inspection. That lowers the amount of guesswork left in the runtime. It also leaves the next step obvious: prove the current tip live, then decide how much of the new capability substrate should be wired into actual runtime decisions rather than just being defined cleanly on paper.
