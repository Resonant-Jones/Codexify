#   Dev Log - 2026-04-20

## Summary

runtime boundary hardening day

Today was mostly about making the runtime more explicit about its own limits. The work tightened retrieval boundaries, bounded the tool-augmented completion path, and split candidate traces into pure normalization plus derived graph-candidate handling. The common thread was simple: fewer implied behaviors, more enforceable contracts.

That matters because Codexify gets more trustworthy when identity, widening, tool execution, and diagnostic artifacts all have visible rules instead of being inferred from side effects.

## What I completed

### Retrieval boundary hardening

- guardian/context/broker.py now enforces strict user_id isolation at the aggregation boundary and requires widening to carry an explicit reason.
- docs/architecture/router-decision-table.md and docs/architecture/00-current-state.md now state that retrieval results stay user-bound and that no-widening normalizes to none.
- The retrieval tests now cover cross-user exclusion, no-widening behavior, and the case where widening is allowed but must be attributed to a concrete reason.

### Bounded tool-augmented completion

- guardian/core/ai_router.py, guardian/core/chat_completion_service.py, guardian/protocol_tokens.py, and guardian/workers/chat_worker.py now support a bounded one-turn command-bus path.
- If the model emits a structured tool decision, the service executes exactly one command-bus invoke, reinjects the result, and requests one final assistant answer.
- The runtime now surfaces messageId, requestId, toolTurnId, toolTurnState, loopStopReason, and commandRunId so the tool path is visible instead of implied.
- docs/architecture/flows.md, docs/architecture/system-overview.md, and docs/architecture/tech-debt-and-risks.md were updated to describe the bounded slice.
- The tests now prove plain completions bypass the bus, tool decisions execute once, and the worker forwards the observability fields.

### Candidate trace normalization and graph candidate mapping

- guardian/core/candidate_normalizer.py introduces a pure normalization layer for candidate_trace payloads.
- guardian/workers/candidate_ingest_worker.py now normalizes candidate traces inside the ingest worker without persisting anything.
- guardian/memory_graph/graph_candidate_mapper.py turns normalized entities into derived graph candidates with dedupe and scope-derived relationships, still without persistence.
- The candidate-trace route and worker tests keep this path separate from canonical chat messages.

### Auth/runtime audit after DB reset

- docs/architecture/2026-04-20-auth-runtime-mode-audit.md records the read-only audit after a database wipe.
- The audit found two Alembic heads and flagged that a fresh bootstrap still needs a merge migration.
- It also documented the debug/dev-mode X-User-Id conflation risk, while confirming that single-user bootstrap still coheres around the local user.

## Important outcome

The biggest shift is that the system is no longer relying on implied behavior at the seams. Retrieval now has an explicit user boundary and explicit widening reasons. Tool use now has a bounded execution story instead of an open-ended loop. Candidate traces now have a deterministic normalization path and a graph-candidate seam without pretending to be canonical chat state.

That matters because the runtime can now explain what it did, what it widened, and what it refused to persist. The day was mostly about making the system more truthful, and that is a real stability improvement even when the work stays at code/test level.

# Narrative Log

Dev Log - 2026-04-20

Today was a boundary day. Not a flashy one. The kind that matters because it removes places where the system can quietly tell the wrong story about itself.

The first correction was retrieval. The broker stopped treating user_id as a soft convention and started enforcing it at the aggregation boundary. Widening is no longer just something that happens when retrieval gets loose; it now carries a reason, and when there is no widening the state normalizes to none. That is a small change on paper, but it closes a common failure mode in local-first systems: cross-user bleed that only shows up when someone tries to explain a result after the fact.

Then the completion path got the same treatment. The model can still answer plainly, but if it emits a structured tool decision, the runtime now does one command-bus invoke, reinjects the result, and stops after one bounded turn. The important part is not that tool use exists. The important part is that the turn now has a shape the system can name: a toolTurnId, a toolTurnState, a loopStopReason, and a commandRunId. That turns tool execution from an implicit side effect into something the worker and the logs can actually describe.

Candidate traces followed the same pattern. They were split into a pure normalization layer and a derived graph-candidate seam. The ingest worker still does not persist canonical state, which is the right call here. Candidate traces are useful runtime evidence, but they are not truth by themselves. Keeping them log-only preserves that boundary and leaves room for later graph work without smuggling in a persistence model too early.

The day ended with a read-only audit after a database reset, and that audit was useful precisely because it was unsentimental. It found a multi-head Alembic state that still needs a merge migration, and it called out how debug-mode X-User-Id handling can still blur display identity with persistent ownership. Neither of those findings is glamorous, but both matter if the system is meant to survive resets without losing its bearings.

So the shape of the day was this: retrieval became more honest, the tool loop became bounded, candidate traces became deterministic, and the bootstrap story got another hard look. That is not the kind of work that looks dramatic in a diff, but it is the work that makes the next validation pass narrower and the next failure easier to name.
