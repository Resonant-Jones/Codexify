# Daily Log — April 8, 2026

Today was about making a boundary honest.

The personal facts area in Settings stopped being a static explanation and became a real lifecycle surface. The dock gained a Personal Facts tab, the panel was reorganized around candidates, verified facts, and history, and the copy was tightened so the product says what it means. Candidates are quarantined. Verified facts are runtime-eligible. History is a ledger. That sounds subtle, but it matters: it turns the page from decorative UI into an explicit trust model.

The second commit made that trust model operational. A new usePersonalFacts hook and matching API helpers now fetch live data, pull detail, evidence, and revision records, and drive the approve, dispute, amend, and retire actions. The panel refreshes after mutations, tracks per-fact busy state, and handles loading and error cases instead of leaning on fixed sample data. In other words, the surface now talks to the backend as a state machine rather than as a storyboard.

The tests followed the same shift. They were expanded to cover live-data rendering, section switching, evidence disclosure, edit flows, and empty states. The dock tests were also updated to reflect the sticky layout and the new tab. That is not just wider coverage. It is the codebase admitting that this surface has behavior worth protecting.

What changed underneath the surface is the system's honesty about identity data. Personal facts now has a visible lifecycle and a visible boundary between quarantined evidence and runtime-trusted facts. That makes drift harder to hide and makes future decisions about promotion or exclusion much cleaner.

The release contract has not changed with it. The current-state doc still treats personal_facts as quarantined from the supported beta surface, so this should be read as implementation hardening, not as a claim of release inclusion.
