Summary

runtime, packaging, and boundary hardening day

Most of the work today was about making Codexify harder to misread. Identity modes are now explicit, graph writes stay derived, launcher state is materialized instead of inferred, and diagnostics have a backend-only place to live. The thread running through all of it was reducing ambiguity before it leaks into runtime behavior.

This was not a flashy feature day. It was a day of tightening the contract: who owns what, what is canonical, what is derived, what is only diagnostic, and what the packaged app should do when it boots on a real machine. That kind of work matters because it removes hidden assumptions from the path before they turn into support pain.

What I completed

Runtime mode and account boundaries

- Merged and documented the single_user / multi_user split.
- Made account ownership, retrieval scoping, and exportability explicit so user_id is treated as a hard boundary in multi_user.
- Updated the current-state truth so the short-horizon runtime contract reflects those invariants.

Launcher startup state and package hygiene

- Materialized the packaged launcher startup-state file so first launch and setup handoff are explicit instead of guessed.
- Hardened setup-wizard dependency detection so it prefers an explicit custom path, then PATH, then macOS fallbacks, and says what it found.
- Tightened bundle staging so broken symlinks and symlinked directories are skipped rather than poisoning the packaged resource tree.

Memory graph derivation

- Added a non-blocking graph-write hook with idempotent candidate emission.
- Kept graph candidate generation derived from canonical Postgres rows instead of making graph persistence part of completion success.
- Canonicalized the memory-graph indexing plan and schema notes so the graph reads as an indexing layer, not a second source of truth.

Diagnostic runtime surfaces

- Added the backend-only candidate_trace surface for pre-answer completion candidates.
- Kept it attempt-scoped, ephemeral, export-excluded, and intentionally non-canonical.
- Added route coverage so the surface behaves like a diagnostic tool rather than a mutation path.

Important outcome

Codexify is no longer treating identity as an implicit background assumption. It now has named runtime modes and a boundary model that the docs and request handling can enforce.

The system also stopped conflating derived or optional work with canonical state. Graph ingestion can now exist as a clean seam, launcher state is written explicitly for the packaged path, and candidate_trace can explain what the completion pipeline considered without pretending to be chat history.

Narrative Log

Dev Log - 2026-04-19

This was a day of narrowing the system until it described itself more honestly. Identity, packaging, derived state, and diagnostics all got a little less ambiguous, which is the sort of progress that is easy to miss if you only look for visible features.

The first thread was runtime truth. The single_user and multi_user split is now named in the architecture, and the account boundary is no longer an implied convention. In multi_user, user_id is supposed to be enforced everywhere that matters, including retrieval and exportability. That matters less as a slogan than as a guardrail: once a system starts mixing ownership assumptions into request handling, later features inherit ambiguity they never asked for. Today’s doc work makes that ambiguity harder to reintroduce.

The second thread was the launcher path. The packaged app now writes a startup-state file and uses it as an explicit handoff point between bootstrap and the local runtime. That is a small thing mechanically, but it is the right shape for a desktop system that has to survive first launch, setup, and recovery without guessing what state it is in. Alongside that, setup-wizard dependency detection became more truthful about what it found, and bundle staging stopped trusting broken symlinks or symlinked directories. The result is less optimism baked into packaging.

The third thread was the memory-graph seam. The new graph-write hook is deliberately boring in the right way: derived from canonical rows, non-blocking, idempotent, and subordinate to the chat worker. That is the correct shape if the graph is meant to enrich the system later without becoming a second persistence authority now. The indexing plan and schema notes put the same idea into words: the graph is an index, not a rival truth. That distinction matters because the wrong abstraction here would quietly turn optional enrichment into a durability dependency.

Then candidate_trace gave the completion pipeline a place to speak without becoming canonical. It exposes pre-answer candidates for operators and tests, but it stays backend-only, attempt-scoped, and export-excluded. That separation is the whole point. The runtime can now show its working without blurring the line between evidence and state.

By the end of the day, the system felt more explicit in a way that should age well. The runtime knows more clearly who owns what, the graph layer knows it is derived, the packaged launcher is less brittle around startup handoff and filesystem artifacts, and the completion pipeline can expose a little more of its internal motion without pretending that motion is durable truth. That leaves the next layer of work with fewer hidden assumptions to trip over.
