Summary

runtime truth-alignment day

This was a day about removing false certainty from the desktop shell and the retrieval surfaces. The work made startup state explicit, gave the app a recovery path when the local runtime is unavailable, and turned retrieval posture into something the command center can compare and explain instead of just display.

What I completed

Launcher handoff and runtime recovery

- Added a structured launcher startup handoff bridge in the desktop runtime layer, with explicit statuses for setup incomplete, runtime unavailable, ready handoff, and launcher unavailable.
- Wired the app to wait on that routing decision before it chooses the startup path.
- Added a recovery surface so the shell can send the user back to bootstrap instead of pretending a broken startup is a normal runtime state.

Command Center retrieval posture

- Added comparison and explanation logic so the latest retrieval posture can be compared against the previous one across source mode, boundary label, retrieval override, widen reason, and conversation-only state.
- Surfaced that comparison in the command center as a compact strip with readable changed-field explanations.
- Tightened scroll containment so the trace workbench and event console stay bounded and behave like a controlled surface instead of stretching the viewport.

Persona Studio and mobile interaction

- Reworked Persona Studio into a two-lane layout, separating configuration from the ephemeral harness so the draft-testing surface reads as its own lane.
- Added a global mobile press-feedback layer and tap-target contract, including reduced-motion handling, press scaling, and navigation-rail behavior.
- That makes the mobile shell feel less ad hoc and reduces subtle interaction drift across buttons, menus, and share actions.

Architecture atlas and supported-path truth

- Refreshed the architecture atlas so the ADR lane sits in the official reading order after current truth and validity, before the diagram packs.
- Added the ADR hub and index docs to formalize the decision history behind completion acceptance, dual state machines, message/request identity, and retrieval policy.
- Corrected `LOCAL_CHAT_MODEL` in the environment example to match the available Ollama model name and updated the current-state notes to reflect the fix while keeping live verification pending.
- Refreshed the supported-path proof artifact so the current blocker set is explicit rather than implied.

Important outcome

The shell is no longer treating startup as a binary guess. It now has named states for setup, handoff, and recovery, which means failure can be shown instead of hidden.

The command center moved in the same direction. Retrieval posture is now something the operator can compare and explain, not just inspect after the fact. That shifts it from raw telemetry toward operational truth.

The architecture docs also became more honest. The ADR lane now exists as a first-class reading path, which matters because the system's decision history is part of the product surface, not an external notebook.

Narrative Log

Dev Log - 2026-04-14

This was a day about replacing guesses with states. The desktop shell used to treat startup as a mostly implied path: if the runtime was there, things moved forward, and if it was not, the user experience had too little vocabulary for the difference between setup, handoff, and recovery. That changed. The launcher handoff is now structured, the app can wait for a routing decision, and there is a recovery surface when the local runtime is configured but not actually ready. That is not decorative work. It is the difference between a shell that assumes success and a shell that can tell the truth when startup is incomplete.

The command center moved in the same direction. Retrieval posture stopped being something you had to infer from trace fragments and became something the UI can compare, explain, and annotate against the previous run. That is a small surface change and a larger operational one. The system can now describe when retrieval shifted, what kind of shift it was, and which part of the posture moved. At the same time, the containment fixes made the trace workbench and event console behave like bounded parts of a single surface instead of drifting into a loose, scroll-heavy composition.

Persona Studio was tightened for a similar reason. The ephemeral harness is now separated from the configuration lane, which makes the draft-testing role legible instead of incidental. The mobile shell picked up a global press-feedback layer and stronger tap-target behavior, which is the kind of work that does not advertise itself but reduces friction everywhere the interface is touched. These changes are small in isolation, but they all point in the same direction: the UI should express the system's state cleanly, not obscure it behind layout noise.

Underneath the UI work, the architecture corpus was brought back into alignment with how the system is actually being reasoned about. The atlas now routes readers through an ADR lane, and the new decision docs make the acceptance model, state separation, identity boundary, and retrieval policy explicit. That matters because these are not side notes. They are the system's operating assumptions, and they should be readable as history instead of inferred from implementation detail. The runtime proof notes also stayed honest: the local model name was corrected to match the Ollama registry, but the populated retrieval-posture seam still needs to be wired before the live path is complete.

What this makes possible next is straightforward. The remaining work is to rerun the exact-main proof, close the retrieval-posture emission gap, and validate the launcher handoff and recovery path under the packaged shell. After that, the system is less about inventing new behavior and more about keeping the behavior it already claims to have honest.
