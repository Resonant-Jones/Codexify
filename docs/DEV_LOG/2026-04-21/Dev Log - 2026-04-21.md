Boundary-hardening and observability day.

What I completed

Graph and trace truth

- Candidate ingest now inspects graph candidates without persisting them, which keeps the inspect path separate from the write path.

- 00-current-state.md now says graph is still enrichment-only, so the trace fields describe present reality instead of hinting at active graph retrieval.
- Tests now cover source-mode fallback, explicit overrides, latest-turn trace propagation, and payload-summary forwarding.

Bounded tool proof

- Added a live runtime proof for the bounded tool-augmented completion slice on the supported Compose stack.
- The proof covers the plain-answer path, one bounded tool turn, the hard stop after one tool turn, and the blocked-command failure path.
- This is the first item in this batch that crossed from code/test evidence into live runtime evidence.

Extension admission and registry

- Install approval now runs through an explicit gate that records a decision before anything is promoted into the capability registry.
- Approved permissions must stay within the requested permission set, which makes the boundary auditable instead of implied.
- Export and restore now carry the install and registry state, so extension authorization survives persistence instead of being reconstructed from memory.
- The new seams are covered by approval, rejection, owner isolation, and scope-filter tests.

Runtime visibility

- Added a read-only desktop runtime monitor for backend health surfaces and launcher artifacts, with bounded statuses instead of a binary healthy/unhealthy flag.
- Added a runbook that explains what the monitor does and does not prove, which matters because runtime truth is split across HTTP surfaces and on-disk artifacts.

UI, storage, and cleanup

- Flow Builder now reads as a real three-pane workspace with a chat dock, graph canvas, and parameter rail.
- Persona Studio was aligned with the approved mockup, then refined, and media uploads now survive container rebuilds because the storage path was canonicalized.
- The dead frontend tools shim adapter was removed, and the command-bus tests now exercise the direct seam instead of the old adapter layer.

Important outcome

The system stopped treating graph work as a vague future feature and started treating it as a separate phase with its own truth boundaries. That matters because it keeps enrichment from being mistaken for retrieval and makes it easier to say what graph can do today without overselling it.

Extension installation became auditable in the same way. Approval, registry promotion, and export/restore now share a record trail, so capability admission is a decision process instead of a loose codepath.

The runtime side also got more trustworthy. There is now one live bounded-tool proof on the supported stack, plus a monitor that distinguishes ready, degraded, not ready, unreachable, and missing artifact conditions instead of collapsing them into noise.

Narrative Log - 2026-04-21

Today was a boundary-hardening day. The useful work was not adding another broad capability, but making the existing ones more honest about what they are doing. The bounded tool loop got a live proof on the supported Compose stack, and the surrounding trace machinery was tightened so the runtime can be read back without guesswork.

The graph work followed the same pattern. Candidate ingest now inspects graph candidates without persisting them, and the chat completion service now carries graph-ready trace placeholders instead of leaving the payload to imply more than it knows. That matters because the system can now say, in plain terms, that graph is still enrichment-only. It is prepared for, but not yet active in retrieval.

The extension pipeline became more deliberate as well. Install approval now goes through an explicit gate, approved permissions must remain a subset of what was requested, and registry promotion records provenance rather than just accepting an opaque install. Export and restore now carry that same state. The effect is small but important: extension capability is turning from an assumption into a decision trail.

On the operator side, a read-only desktop runtime monitor was added so the backend health surfaces and launcher artifacts can be checked without collapsing them into a single vague status. The runbook makes the same point: runtime truth is split across multiple surfaces, and the tool should preserve that split instead of hiding it. That pairs well with the live proof work, because the proof and the monitor now speak the same bounded vocabulary.

The product shells were also cleaned up so they match the backend more honestly. Flow Builder now reads as a three-pane workspace instead of a collapsed surface. Persona Studio was brought back in line with the approved mockup, then refined, and its media uploads now survive container rebuilds because the storage path was normalized. A dead frontend tools shim adapter was removed too, which is the kind of cleanup that only matters when it stops a system from pretending there is a layer where there is none.

The deeper change is simple: Codexify is getting harder to misunderstand. Some of that is visible in the UI, but most of it is in the contracts underneath it. Graph work has a shape. Extension installs have a decision record. The runtime can be monitored without guesswork. That makes the next round of work easier because the system is no longer hiding its boundaries from us.
