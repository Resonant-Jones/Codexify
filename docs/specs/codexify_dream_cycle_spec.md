# Codexify Dream Cycle Spec

## Status

**Classification:** Product / architecture concept spec  
**Stage:** Pre-ADR record  
**Scope:** Thread linking, session consolidation, daily reports, idle-time batch work, memory backfill, and co-work automations  
**Non-goal:** This spec is not an implementation task and does not change current beta release scope.

---

## 1. Purpose

Codexify should support a controlled background consolidation layer that reviews user activity after active work sessions and converts the day’s raw traces into useful, inspectable continuity artifacts.

The core idea is simple:

> Codexify should capture lightly while the user is working, then consolidate deeply when the user is idle.

This allows the system to use local compute resources during overnight or low-activity windows without slowing down the foreground chat experience.

The Dream Cycle is intended to help users answer questions like:

- What was I working on yesterday?
- Which thread was that idea in?
- What did we finish?
- What reached a satisfactory save point?
- What did we leave unresolved?
- Which commits, docs, threads, and artifacts belong together?
- What should be documented, summarized, reviewed, or backfilled?

---

## 2. Core Thesis

Codexify should operate in two tempos.

### 2.1 Waking Capture

During active use, Codexify should perform lightweight tagging and evidence capture only.

Foreground work should prioritize responsiveness. The system should avoid heavy reflection, expensive summarization, or broad memory mutation during active interaction unless explicitly requested.

### 2.2 Sleeping Consolidation

During idle windows, especially overnight in 24/7 local server mode, Codexify should run batch consolidation jobs that review the day’s activity and produce structured continuity outputs.

These outputs should be reviewable, provenance-linked, and reversible.

---

## 3. Definitions

### Dream Cycle

The overall local-first background reflection system that reviews recent work, links related material, identifies open loops, proposes memory updates, and drafts follow-through work.

### Waking Tags

Lightweight real-time markers attached during active work. These tags flag candidate events or relationships for later review without doing heavy processing immediately.

### Nightly Synthesis

The idle-time batch process that consumes Waking Tags, daily reports, thread activity, commits, changed files, task events, and artifacts to produce continuity outputs.

### Morning Brief

A user-facing summary generated after Nightly Synthesis. It explains what happened, what changed, what remains open, and what Codexify recommends reviewing.

### Savepoint Ledger

A structured list of work areas that appear to have reached a stable stopping point.

### Open Loop Ledger

A structured list of work areas that appear unresolved, abandoned, interrupted, or waiting on follow-through.

### Thread Constellation

A graph-like relationship layer connecting threads, messages, artifacts, commits, daily reports, files, tasks, and generated documents.

### Memory Patch Queue

A reviewable queue of proposed memory updates generated by consolidation. These proposals must not automatically become durable identity memory without policy approval.

### Co-work Automation

A guarded automation that performs helpful follow-through based on detected evidence patterns, such as drafting documentation after code changes or summarizing commits after a long work session.

---

## 4. Product Goals

Dream Cycle should make Codexify feel like a system that preserves continuity across days, not just across messages.

It should:

- help the user recover yesterday’s context quickly
- connect scattered but related threads
- identify unresolved work without requiring manual tracking
- distinguish finished work from paused work
- detect documentation and follow-through gaps
- convert repeated workflows into candidate skills or playbooks
- propose memory backfills intentionally rather than mutating identity silently
- use idle local compute productively
- remain inspectable and trustworthy

---

## 5. Non-Goals

Dream Cycle is not:

- a hidden autonomous agent that silently rewrites user memory
- a replacement for the current chat runtime
- a beta-release blocker
- a full workflow engine in V1
- a general-purpose self-modifying agent harness
- a license to mutate identity traits without user consent
- a reason to collapse project memory, thread memory, and identity memory into one bucket
- a substitute for explicit user review where review is required

---

## 6. Conceptual Architecture

```text
Active work session
  -> foreground chat, docs, commits, artifacts, task events
  -> lightweight Waking Tags
  -> end-of-day Daily Report / Work Index
  -> Nightly Synthesis job
  -> continuity artifacts
  -> Morning Brief
  -> optional reviewed memory/doc/task/skill patches
```

Dream Cycle should operate as a derived inspection and consolidation layer over existing runtime facts.

It must not replace the canonical system of record for threads, messages, documents, artifacts, task events, or audit logs.

---

## 7. Input Sources

Dream Cycle may consume:

- chat threads
- chat messages
- thread metadata
- project metadata
- Codex entries
- thread-linked artifacts
- generated documents
- uploaded documents
- daily audit reports
- git commits
- changed file lists
- task events
- retrieval traces, when available
- post-completion eval summaries, when available
- user-authored tags and flags
- existing memory entries
- personal facts and evidence rows, subject to policy
- workspace scratchpad entries, if enabled

Each consumed source must preserve provenance.

## 8. Output Artifacts

Dream Cycle may produce the following derived artifacts.

### 8.1 Morning Brief

A human-readable daily summary.

Recommended sections:

- Yesterday’s headline
- Work completed
- Work paused at a save point
- Open loops
- Important threads
- Important commits
- Documentation gaps
- Suggested next actions
- Memory proposals requiring review

### 8.2 Daily Work Index

A structured machine-readable index for the day.

Possible fields:

```json
{
  "date": "2026-04-27",
  "threads": [],
  "commits": [],
  "filesChanged": [],
  "artifactsCreated": [],
  "openLoops": [],
  "savepoints": [],
  "docGaps": [],
  "memoryCandidates": [],
  "skillCandidates": []
}
```

### 8.3 Thread Link Suggestions

Candidate relationships between threads, artifacts, commits, documents, and reports.

These should include:

- source object
- target object
- relationship type
- confidence
- evidence excerpts or metadata
- reason for suggestion

### 8.4 Open Loop Ledger

A structured set of unresolved work items.

Examples:

- “User asked whether X was finished, but no final savepoint was recorded.”
- “Implementation landed, but docs follow-through was not observed.”
- “A task was discussed as next step, but no commit or task artifact links to it.”

### 8.5 Savepoint Ledger

A structured set of work areas that appear safely paused.

Examples:

- tests passed and commit landed
- user confirmed design direction
- documentation updated
- no active blocker marker remains
- latest daily report marks the area as complete or green

### 8.6 Memory Patch Proposals

Proposed updates to memory or identity-related records.

These should be separated by target layer:

- thread memory
- project memory
- workflow preference
- durable identity memory
- diary-only note
- excluded-from-modeling note

Identity-related proposals must remain review-first.

### 8.7 Skill Patch Proposals

Reusable procedures observed during work.

Examples:

- “When generating Codexify implementation tasks, use the two-lane task protocol.”
- “When updating shell layout, check AppShell posture tests and Documents interaction tests.”
- “When validating beta readiness, read supported-profile, health, catalog, and live completion evidence together.”

### 8.8 Co-work Automation Drafts

Generated follow-through artifacts requiring review.

Examples:

- docs delta draft after code refactor
- release-note draft after multiple commits
- test checklist after UI behavior changes
- morning task list from unresolved thread references
- summary of what changed in a selected subsystem

## 9. Waking Tag Taxonomy

Initial candidate tags:

| Tag | Meaning |
|---|---|
| `thread.reference.detected` | A message refers to another thread, report, artifact, or commit. |
| `idea.open_loop` | A question, task, or decision appears unresolved. |
| `savepoint.possible` | Work may have reached a stable stopping point. |
| `doc_gap.possible` | Code changed without matching documentation. |
| `test_gap.possible` | Code changed without matching test evidence. |
| `memory.candidate` | A fact or preference may deserve memory review later. |
| `skill.candidate` | A reusable workflow may have emerged. |
| `thread_link.candidate` | Two or more threads may belong to the same conceptual cluster. |
| `artifact_link.candidate` | An artifact may be related to a thread, commit, or report. |
| `daily_report.anchor` | The daily report should be treated as a timeline anchor. |
| `followthrough.candidate` | The system should consider proposing a next action. |
| `identity_sensitive.review_required` | A candidate memory may touch identity-sensitive material and must not be auto-applied. |

Tags should be canonicalized before they spread across the codebase.

---

## 10. Relationship Types

Initial candidate relationship vocabulary:

| Relationship | Meaning |
|---|---|
| `seeded_by` | Artifact or memory was created from a source message/thread. |
| `references` | Object directly references another object. |
| `continues` | Thread or artifact continues work from another source. |
| `blocks` | Object describes a blocker for another work item. |
| `resolves` | Object appears to resolve an open loop. |
| `documents` | Object documents code, feature, or decision. |
| `implements` | Commit or task implements a described change. |
| `tests` | Commit, artifact, or report validates behavior. |
| `supersedes` | Newer artifact replaces an older one. |
| `related_to` | Weak relationship that should not imply causality. |
| `savepoint_for` | Object marks a stable stopping point for a work area. |
| `open_loop_for` | Object identifies unresolved work for a work area. |

These relationship names should be treated as contract-bearing tokens if implemented.

## 11. Scheduling Model

Dream Cycle should support both automatic and manual execution.

### 11.1 Automatic Nightly Run

Recommended default:

- run during configured sleep/idle window
- skip or defer if active user interaction is detected
- apply compute budget limits
- write a run summary even if partially skipped

### 11.2 Manual Run

The user should be able to run Dream Cycle manually for:

- today
- yesterday
- a selected date range
- a selected project
- a selected thread
- a selected release window

### 11.3 Idle Opportunistic Runs

Optional later behavior:

- short consolidation passes during long idle periods
- low-priority background indexing
- thread-link candidate generation when local compute is available

---

## 12. Memory and Identity Policy

Dream Cycle must follow Codexify’s identity boundary principles.

### 12.1 Separate What Happened From Who the User Is

Raw chat, commits, reports, and work traces describe what happened. They do not automatically define durable identity.

### 12.2 Review Required for Identity Memory

Any proposal that would alter durable user identity, values, preferences, or long-term traits must enter a review queue.

### 12.3 Project Memory May Be More Automatic

Project facts, file conventions, build commands, test workflows, and repo-specific patterns may be promoted with fewer restrictions, but should still preserve evidence and allow rollback.

### 12.4 Diary and Exclusion Rules Must Be Honored

Threads marked diary or excluded from modeling must not be used for memory promotion. If included in raw search by user choice, they should remain clearly bounded.

## 13. UX Surfaces

### 13.1 Morning Brief

A daily review card or page.

Recommended actions:

- accept memory proposal
- dismiss memory proposal
- open source thread
- open source commit
- create task
- draft docs
- mark as save point
- mark as still open
- link threads
- unlink suggested relationship

### 13.2 Thread Constellation View

A relationship map showing how threads, commits, artifacts, and reports connect.

This should be optional and not clutter the main chat surface.

### 13.3 Open Loop Ledger

A focused list of unresolved work across projects and threads.

Possible filters:

- project
- thread
- date
- confidence
- source type
- user-reviewed status

### 13.4 Savepoint Ledger

A focused list of stable stopping points.

Useful for answering:

- “Where did I leave off?”
- “What can I safely ignore today?”
- “What was actually finished?”

### 13.5 Memory Patch Queue

A review interface for proposed memory changes.

Must show:

- proposed memory text
- target memory layer
- source evidence
- confidence
- sensitivity classification
- apply / edit / reject actions

## 14. Example: Documentation Gap Co-work Automation

### Input Evidence

- code commits changed runtime or frontend behavior
- no docs changed in the same window
- daily report flags architecture or UI state movement
- thread conversation included design rationale

### Detection

```text
IF commits include code changes
AND changed files match behavior-bearing paths
AND no documentation files changed
THEN emit doc_gap.possible
```

### Dream Action

Nightly Synthesis can:

1. read relevant commit metadata
2. inspect changed file paths
3. locate related thread messages
4. produce a documentation delta draft
5. link the proposed doc update back to source commits and threads
6. ask user to approve before applying

### Output

A Co-work Automation Draft:

```text
Detected possible documentation gap:
- frontend shell behavior changed
- documents sidebar behavior changed
- no matching docs update detected

Suggested follow-through:
- update workspace/sidebar behavior notes
- preserve distinction between UI posture and runtime truth
- cite commits and source thread
```

## 16. Runtime Constraints

Dream Cycle should respect current runtime boundaries.

- Background processing must not interfere with active chat completion.
- Heavy consolidation should run at low priority.
- All writes must be durable and auditable.
- Partial runs must be recoverable.
- Failed runs must report failure reasons.
- Identity-impacting outputs must require review.
- Generated artifacts must preserve source lineage.
- Export/restore must preserve Dream artifacts and relationships if they become durable product state.

---

## 17. Safety and Trust Rules

Dream Cycle must never:

- silently mutate durable identity memory
- invent thread relationships without marking confidence
- hide source evidence
- overwrite user-authored notes without approval
- treat speculation as fact
- collapse “related” into “caused”
- expose diagnostics inside the primary chat loop by default
- run expensive jobs while the user is actively interacting, unless configured
- count derived findings as release truth without proof

Dream Cycle should always:

- preserve provenance
- show confidence
- make review state explicit
- let users accept, edit, reject, or defer findings
- distinguish raw evidence from generated interpretation
- keep project memory, thread memory, and identity memory separate

## 18. Suggested Implementation Phases

### Phase 0: Record the Spec

Create this spec as a pre-ADR record. No runtime change.

### Phase 1: Daily Work Index

Parse daily audit reports into a structured local index.

Outputs:

- daily report entity
- commit list
- changed files
- pass/warn/fail summary
- source links

### Phase 2: Thread–Artifact Relationship Proposals

Extend lineage to suggest relationships between:

- thread ↔ message
- thread ↔ artifact
- thread ↔ commit
- daily report ↔ thread
- daily report ↔ commit

### Phase 3: Waking Tags

Emit lightweight candidate tags during active work.

No heavy consolidation yet.

### Phase 4: Nightly Dream Job

Add scheduled idle-time consolidation.

Outputs:

- Morning Brief
- Open Loop Ledger
- Savepoint Ledger
- thread-link suggestions
- doc/test gap findings

### Phase 5: Memory Patch Queue

Generate reviewable memory proposals.

Identity memory remains review-first.

### Phase 6: Co-work Automation Drafts

Generate reviewable follow-through drafts for documentation, release notes, task summaries, and workflow updates.

### Phase 7: User-Facing Dreaming Feature

Expose Dream Cycle as a polished product surface.

Possible product language:

> Codexify reviewed yesterday’s work while you were away. Here’s what it found.

## 19. Open Questions

1. Where should Dream Cycle live in the UI: Dashboard, Workspace, Diagnostics, or its own Morning Brief surface?
2. Should Dream findings be part of export/restore V1 immediately or only after the model stabilizes?
3. What is the minimum viable relationship graph before Neo4j becomes useful?
4. Should git commits be ingested directly, or should daily reports remain the first structured bridge?
5. What confidence threshold should allow automatic project-memory updates?
6. What findings require explicit user review?
7. How should Diary threads interact with daily summaries?
8. Should Co-work Automations create drafts only, or eventually open implementation tasks?
9. Should Dream Cycle be disabled by default until the user opts into local idle processing?
10. How should compute budgets be configured for laptops vs always-on desktops?

---

## 20. Decision Bias

Start conservative.

The first useful Dream Cycle does not need autonomy. It needs continuity.

The first version should answer:

> What happened, what matters, what connects, and what needs review?

If Codexify can answer those four questions every morning with strong provenance, it will feel meaningfully more alive without becoming unsafe or noisy.

---

## 21. Final Principle

Dream Cycle should make Codexify reflective, not presumptive.

It should help the user recover context and notice patterns, while preserving the user’s authority over memory, identity, and action.
