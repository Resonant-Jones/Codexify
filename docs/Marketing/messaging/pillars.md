# Messaging Pillars and Proof Mapping

## Pillar 1: Truthful Runtime Boundaries

Claim direction:
Codexify surfaces what is implemented versus what is only planned.

Allowed claim examples:

- release posture is tied to dated evidence artifacts
- current-state docs constrain short-horizon claims

## Pillar 2: Local-First Operational Control

Claim direction:
Codexify prioritizes local, inspectable execution over opaque central orchestration.

Allowed claim examples:

- supported local paths are explicitly documented
- operator runbooks and proof scripts are part of release truth

## Pillar 3: Identity and Governance Discipline

Claim direction:
Identity scope and policy are treated as infrastructure.

Allowed claim examples:

- boundaries are stated in contracts and routes
- governance claims remain evidence-bound and non-magical

## Pillar 4: Failure Visibility and Drift Prevention

Claim direction:
Codexify makes failure states and drift visible to operators.

Allowed claim examples:

- known risks are tracked in named artifacts
- daily/phase audits preserve continuity of operational truth

## Proof Tier Mapping Rules

- `implemented`: backed by code-level or campaign/task implementation evidence
- `verified`: backed by deterministic tests or validation commands
- `live-proven`: backed by dated runtime proof artifacts on supported paths

A claim must use the highest tier that can be justified by evidence, never higher.
